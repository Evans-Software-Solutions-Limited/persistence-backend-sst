## What the M3 backend looks like now (post-merge)

### Endpoints

```
GET     /sessions?status=in_progress&limit=1   ← resume detection
GET     /sessions/:id                          ← detail with nested exercises
GET     /sessions                              ← list (paginated)
POST    /sessions                              ← legacy CRUD create (rarely used now)
PATCH   /sessions/:id                          ← updates (gated PR detection on status=completed)
DELETE  /sessions/:id

POST    /sessions/record                       ← M3 BULK FLUSH (the new path mobile uses on Finish)

POST    /sessions/:id/exercises                ← legacy CRUD (M4 progress edits)
GET     /sessions/:id/exercises                ← read
DELETE  /sessions/:id/exercises/:eid

POST    /sessions/:sid/exercises/:eid/sets     ← legacy CRUD (M4 progress edits)
GET     /sessions/:sid/exercises/:eid/sets     ← read
PATCH   /sessions/:sid/exercises/:eid/sets/:setId
DELETE  /sessions/:sid/exercises/:eid/sets/:setId

GET     /personal-records                      ← user PR catalog (filterable by exerciseId / recordType)
```

### PR detection (server-side, hybrid model)

`PersonalRecordsRepository.recordPRsForSession(userId, sessionId, tx?)`:

1. Loads completed sets joined through session_exercises ⨝ workout_sessions to enforce userId scope.
2. For each completed set with weight + reps: computes Epley 1RM = `weightKg × (1 + reps / 30)`.
3. Upserts into `personal_records` keyed by `(userId, exerciseId, recordType)`. Postgres `ON CONFLICT … DO UPDATE WHERE personal_records.value < excluded.value` — idempotent on replay.
4. Re-syncs `is_personal_record` flag on `exercise_sets`: DEMOTE flagged-but-no-longer-canonical sets, PROMOTE current canonical PR setIds.
5. Scoped to the exercises this session touched. Only `1rm` recordType written (the recordTypeEnum doesn't have `volume` yet).

The `tx?` parameter lets the same method run inside a parent transaction (used by `SessionRepository.recordSession` so PR detection is atomic with the session insert) OR standalone (used by the post-hoc trigger in `sessionsUpdateHandler` for the PATCH-status-to-completed path).

### Bulk-record vs. piecemeal

After PR #48 merges, the M3 mobile flush path is:

1. Mobile keeps the active session in **local state + SQLite** for crash recovery (no per-set network).
2. On Finish (or Discard), mobile builds the full `RecordSessionInput` payload and POSTs **once** to `/sessions/record`.
3. Server inserts session + every exercise + every set + runs PR detection — all in `db.transaction(...)`.
4. Sync queue intent kind: just `recordSession` (single intent per session).

The piecemeal endpoints (`POST /sessions/:id/exercises`, etc.) **stay** for the editing-completed-session use cases that M4 will need.

### Mobile sync queue: what changed

Before #48: 4 chained intent kinds — `createSession` → many `createSessionExercise` → many `createSessionSet` → `updateSession {status: completed}`, in dependency order, with ID swapping.

**After #48: just `recordSession`.** Single intent. The mobile sync worker fires the bulk POST, gets back the canonical session with server-assigned UUIDs, swaps local IDs in SQLite. Much simpler.

If you find yourself implementing chained intent dependencies, you've drifted from the bulk pattern — re-read [`specs/milestones/M3-active-session/BACKEND_BRIEF.md`](../specs/milestones/M3-active-session/BACKEND_BRIEF.md) § 7.

## Final notes

- **Single-Supabase reality.** Both stages target the same DB. Migrations apply on first push to main (via staging-deploy); production-deploy's `db push` is a no-op. A bad migration affects both stages immediately. Treat every migration like a production change.
- **Release-please is wired.** Push to main → "chore: release" PR opens. Merging it publishes a GitHub release → production-deploy + mobile-build-production fire. Don't merge the chore PR until staging is validated.
- **Bulk-record is the future for active sessions.** The piecemeal endpoints stay only for editing completed sessions (M4 / M8 territory). If you find a third use case for them in active session, you're probably re-creating a sync-queue chain — surface it on a PR for review.
