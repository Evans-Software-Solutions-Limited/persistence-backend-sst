# Skill: SST Resource Change

Use this when creating, modifying, or removing AWS resources in the SST infra (API Gateway, Lambda, RDS, S3, etc).

## Before You Start

1. Read the root CLAUDE.md → "Database & Migrations"
2. Understand the current infra layout: `infra/api.ts`, `infra/storage.ts`, etc.
3. Check `sst.config.ts` for how resources are imported and exported

## Common Tasks

### Adding a New Lambda Function

```ts
// infra/api.ts or new infra file
const myFunction = new aws.lambda.Function("MyFunction", {
  role: apiRole,
  handler: "microservices/core/src/index.handler",
  timeout: 30,
  environment: {
    DATABASE_URL: db.connectionString,
  },
});
```

### Adding Environment Variables

- Use SST secrets: `sst secret set KEY value`
- Injected into Lambda via `Resource.XyzUrl?.value` or `process.env.XYZ`
- Do not hardcode secrets in code

### Database Resource Changes

- Neon is external (not SST-managed in code)
- DB URL: injected via `sst secret set PersistenceDatabaseUrl <url>`
- Migrations: run via `packages/db/scripts/migrate.ts`

### API Gateway Routes

- Routes defined in Elysia handlers
- API Gateway auto-generated by SST (OpenAPI spec)
- CORS, auth, rate limiting: configure in Elysia or SST resource

## Checklist

- [ ] Resource defined in appropriate file (infra/\*)
- [ ] Environment variables set via `sst secret set`
- [ ] IAM role has correct permissions (principle of least privilege)
- [ ] Timeout appropriate for workload (Lambda: consider cold-start)
- [ ] Resource exports available for other files
- [ ] Removal policy set correctly (dev: destroy on removal; prod: retain)
- [ ] Tests: resource can be provisioned, accessed by handler
- [ ] No hardcoded secrets
- [ ] Conventional commit (infra: ...)

## After You're Done

1. `bun run build` — compiles, no errors
2. Test locally: `bun run dev` — can create/access resource
3. Check `sst.config.ts` — all imports/exports correct
4. Document in root CLAUDE.md if major infra change
